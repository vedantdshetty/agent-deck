1. **settings_panel.go**: Add `ScrollUp()` and `ScrollDown()` public methods to `SettingsPanel`:
   - `ScrollUp()`: decrement `s.cursor` if > 0
   - `ScrollDown()`: increment `s.cursor` if < the settings count upper bound (mirror the bounds logic from the existing key handler's "j"/"down" and "k"/"up" cases)

2. **mcp_dialog.go**: Add `ScrollUp()` and `ScrollDown()` public methods to `MCPDialog`:
   - These should scroll the active scope list cursor (mirror the existing key handler's "j"/"down" and "k"/"up" bounds logic)

3. **global_search.go**: Add a `case tea.MouseMsg` inside the existing `Update()` method (which already accepts `tea.Msg`):
   - `tea.MouseButtonWheelUp`: if `gs.cursor > 0`, decrement cursor and reset `gs.previewScroll = 0`
   - `tea.MouseButtonWheelDown`: if `gs.cursor < len(gs.results)-1`, increment cursor and reset `gs.previewScroll = 0`
   - Return early after handling

4. **home.go**: Add a `case tea.MouseMsg` in `Home.Update()` as a peer to the existing `case tea.WindowSizeMsg` (after WindowSizeMsg, before KeyMsg). The handler must:
   - Check active overlays in priority order before falling through to the main list:
     a. Setup wizard visible: return early (no scroll in wizard)
     b. Settings panel visible (`h.settingsPanel.IsVisible()`): call `h.settingsPanel.ScrollUp()` or `ScrollDown()`, return
     c. Help overlay visible: return early (static content)
     d. Global search visible (`h.globalSearch.IsVisible()`): forward the entire `tea.MouseMsg` to `h.globalSearch.Update(msg)`, return
     e. MCP dialog visible (`h.mcpDialog.IsVisible()`): call `h.mcpDialog.ScrollUp()` or `ScrollDown()`, return. MUST check `h.mcpDialog.IsVisible()` guard before calling scroll methods to prevent unconditional scroll when the MCP dialog is not open.
     f. New dialog, fork dialog visible: return early
   - For the main session list (no overlay):
     - `tea.MouseButtonWheelUp`: if `h.cursor > 0`, decrement `h.cursor`, call `h.syncViewport()`, `h.markNavigationActivity()`, return with `h.fetchSelectedPreview()`
     - `tea.MouseButtonWheelDown`: if `h.cursor < len(h.flatItems)-1`, increment `h.cursor`, call `h.syncViewport()`, `h.markNavigationActivity()`, return with `h.fetchSelectedPreview()`
   - For any other mouse button, return `h, nil` (ignore non-wheel events)

   IMPORTANT: Use `tea.MouseButtonWheelUp` and `tea.MouseButtonWheelDown` (NOT the deprecated `tea.MouseWheelUp`/`tea.MouseWheelDown` constants). Match on `msg.Button`, NOT `msg.Type`.

   IMPORTANT: Handler must be O(1) with no blocking I/O. Preview fetching goes through the existing debounce system via `fetchSelectedPreview()`.

<task type="auto">
  <name>Task 2: Fix light theme dark background bleed-through in preview pane</name>
  <files>internal/ui/home.go</files>
  <action>
Fix the preview pane rendering so that light theme does not show dark background bands from embedded ANSI codes in captured pane content.

**Scope note (UX-02):** The ROADMAP requires light theme correctness in "Codex preview AND live session views." The preview pane is rendered by agent-deck (renderPreviewPane), so ANSI stripping applies there. The "live session view" (when user attaches via Enter) is a direct tmux pane view where agent-deck's TUI is suspended and the user sees the tool's raw output. Agent-deck cannot control ANSI rendering in that context because it does not render the content. The fix here covers the preview pane (agent-deck's rendering path). The same stripANSIBackground helper is also applied to any captured content path used for the preview cache, ensuring all agent-deck-rendered views are clean.

**Approach (try in order):**

1. **Primary fix (ANSI background stripping):** In the preview content rendering section of `home.go` (around line 10092-10122 where captured pane content is rendered), when `GetCurrentTheme() == ThemeLight`, strip background color ANSI escape sequences from each line of the captured content before rendering. Use `github.com/charmbracelet/x/ansi` (already imported at line 10051) to process the content. Specifically, strip `ESC[4xm` family sequences (background colors: `\x1b\[4[0-9]m`, `\x1b\[48;[0-9;]*m`, `\x1b\[49m`) using a regex replacement. Preserve all other ANSI sequences (foreground colors, bold, italic, underline) so formatting is retained.

2. **If approach 1 is insufficient:** Wrap the entire preview content block with a lipgloss style that forces `Background(ColorBg)`. This serves as a fallback base layer under any remaining ANSI.

**Key constraint:** Do NOT strip all ANSI codes. Only strip background color sequences. The goal is to remove dark backgrounds while preserving syntax highlighting and text formatting.

**Where to apply:** Find the section in `home.go` where `renderPreviewPane()` or equivalent function outputs captured tmux pane lines. The fix should apply to all tools (Claude, Codex, Gemini, OpenCode) uniformly whenever light theme is active.

Create a helper function `stripANSIBackground(line string) string` near the preview rendering code that performs the regex replacement. This keeps the fix isolated and testable. The test stub in `light_theme_test.go` (from Wave 0) will verify this function directly.
  </action>
